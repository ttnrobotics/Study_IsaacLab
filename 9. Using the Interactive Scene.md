I’ll explain this tutorial in a **simple and intuitive way**, because this concept (**InteractiveScene**) is **very important in Isaac Lab**, especially for **reinforcement learning and multi-robot simulation**.

---

# 1. The Problem This Tutorial Solves

In previous tutorials, you created the scene **manually** like this:

```python
create_ground()
create_light()
spawn_robot()
spawn_object()
spawn_sensor()
```

If the scene becomes big:

```text
Ground
Lights
Robot
Objects
Camera
Lidar
Table
Boxes
```

Then your code becomes messy.

Example:

```python
spawn_ground()
spawn_light()
spawn_robot()
spawn_box()
spawn_camera()
spawn_lidar()
```

And if you want **100 environments**, you must repeat everything.

Very annoying.

---

# 2. The Idea of InteractiveScene

The **InteractiveScene** solves this problem.

Think of it like a **scene manager**.

Instead of manually spawning things, you **describe the scene once**, and Isaac Lab manages everything.

Conceptually:

```text
InteractiveScene
     │
     ├── Ground
     ├── Light
     ├── Robot
     ├── Objects
     └── Sensors
```

So instead of managing each object yourself, you manage **one scene object**.

---

# 3. What is inside an Interactive Scene?

An InteractiveScene contains **scene entities**.

Entities can be:

### 1️⃣ Non-interactive objects

They don't move or interact.

Examples:

```text
Ground plane
Lights
Walls
Tables
```

---

### 2️⃣ Interactive objects

Objects controlled by physics.

Examples:

```text
Robot
Rigid body
Articulations
```

Example:

```text
Cartpole
Robot arm
Mobile robot
```

---

### 3️⃣ Sensors

Devices that observe the environment.

Examples:

```text
Camera
Lidar
Depth sensor
```

---

So the scene looks like:

```text
InteractiveScene
     │
     ├── Ground
     ├── Light
     ├── Cartpole
     └── Sensors
```

---

# 4. Advantages of InteractiveScene

### 1️⃣ No need to manually spawn assets

Instead of:

```python
spawn_ground()
spawn_light()
spawn_robot()
```

You define them in a **configuration class**.

---

### 2️⃣ Easy to create many environments

For example:

```bash
--num_envs 32
```

This automatically creates:

```text
Env 0 → Cartpole
Env 1 → Cartpole
Env 2 → Cartpole
...
Env 31 → Cartpole
```

Perfect for **reinforcement learning**.

---

### 3️⃣ All objects managed in one place

Instead of managing many variables:

```python
robot
camera
light
objects
```

You just use:

```python
scene
```

---

# 5. Scene Configuration

The scene is defined using a **configuration class**.

```python
class CartpoleSceneCfg(InteractiveSceneCfg):
```

Think of this class as a **blueprint for the scene**.

Example:

```text
Scene Blueprint
     │
     ├── Ground
     ├── Light
     └── Cartpole Robot
```

```python
ground = AssetBaseCfg(
```

Meaning:

```text
Spawn a ground plane
at path:

```

In the USD stage:

```text
World

```

# 7. Light Configuration

```python
dome_light = AssetBaseCfg(
```

Adds lighting.

```text
World
```

# 8. Cartpole Robot

```python
cartpole: ArticulationCfg = CARTPOLE_CFG.replace(
    prim_path="{ENV_REGEX_NS}/Robot"
)
```

Cartpole is a **robot articulation**.

Cartpole system:

```text
Cart
  │
  └── Pole
```

---

# 9. Important Concept: `{ENV_REGEX_NS}`

This is **very important**.

```text
{ENV_REGEX_NS}
```

means:

> "replace with environment name"

Example if `num_envs = 3`

It becomes:

```text
/World/envs/env_0/Robot
/World/envs/env_1/Robot
/World/envs/env_2/Robot
```

So Isaac Lab automatically creates:

```text
World
 └── envs
      ├── env_0
      │     └── Robot
      ├── env_1
      │     └── Robot
      └── env_2
            └── Robot
```

Each environment has **its own robot**.

---

# 10. Creating the Scene

In the main code:

```python
scene_cfg = CartpoleSceneCfg(
    num_envs=args_cli.num_envs,
    env_spacing=2.0
)
```

Meaning:

```text
Create many environments
spacing between them = 2 meters
```

Example layout:

```text
env_0    env_1    env_2    env_3
  |        |        |        |
Cartpole Cartpole Cartpole Cartpole
```

---

Then create the scene:

```python
scene = InteractiveScene(scene_cfg)
```

Now the scene **automatically spawns everything**.

---

# 11. Accessing Scene Objects

You can access objects using their **names from the config**.

Example:

```python
robot = scene["cartpole"]
```

Because in the config we defined:

```python
cartpole: ArticulationCfg
```

So the key is:

```text
"cartpole"
```

---

# 12. Resetting the Scene

Before:

```python
robot.reset()
```

Now:

```python
scene.reset()
```

This resets **everything in the scene**:

```text
Robot
Sensors
Objects
```

---

# 13. Writing Data to the Simulator

Before:

```python
robot.write_data_to_sim()
```

Now:

```python
scene.write_data_to_sim()
```

This sends **all object commands** to the physics engine.

---

# 14. Updating the Scene

Before:

```python
robot.update()
```

Now:

```python
scene.update()
```

This updates:

```text
Robot states
Sensor data
Physics buffers
```

---

# 15. Simulation Loop

The simulation loop now looks like this:

```python
while simulation_app.is_running():

    if reset:
        scene.reset()

    robot.set_joint_effort_target(efforts)

    scene.write_data_to_sim()

    sim.step()

    scene.update()
```

Much cleaner.

# 17. Why This is Extremely Important

This design is used for **Reinforcement Learning**.

Instead of training 1 robot:

```text
1 robot → slow training
```

You train:

```text
1024 robots in parallel
```

So training becomes **1000x faster**.

Example:

```text
env_0  → robot
env_1  → robot
env_2  → robot
...
env_1023 → robot
```
