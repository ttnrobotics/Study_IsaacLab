I’ll explain this tutorial **step-by-step in a very simple way**. The main idea is:

> Instead of creating a new RL environment from scratch, you can **copy an existing one and modify only a few parts**.

In this tutorial, we take the **existing humanoid walking task** and **replace the humanoid robot with the Unitree H1 robot**.

---

# 1. Big Idea of This Tutorial

Original task:

```
Humanoid robot → learn to walk
```

New task we want:

```
Unitree H1 robot → learn to walk
```

But we **don’t want to rewrite the whole environment**.

So we do:

```
Copy existing humanoid environment
↓
Change robot model
↓
Adjust parameters
↓
Register new task
```

---

# 2. The Original Environment

The base environment is inside:

```
isaaclab_tasks.direct.humanoid
```

Main file:

```
humanoid_env.py
```

Inside it there are **two main classes**:

### 1️⃣ Environment configuration

```python
class HumanoidEnvCfg(DirectRLEnvCfg):
```

This contains:

```
robot
simulation
terrain
reward parameters
observation size
action size
```

---

### 2️⃣ Environment implementation

```python
class HumanoidEnv(LocomotionEnv):
```

This actually runs the environment.

Think of it like:

```
HumanoidEnvCfg → settings
HumanoidEnv → environment logic
```

---

# 3. First Step: Copy the File

Instead of modifying the original file (which could break things), we:

Copy:

```
humanoid_env.py
```

Rename it to:

```
h1_env.py
```

Location:

```
isaaclab_tasks/direct/humanoid/
```

Now we have:

```
humanoid_env.py   (original)
h1_env.py         (our new task)
```

---

# 4. Rename the Classes

Inside the copied file we change names:

Original:

```python
HumanoidEnvCfg
HumanoidEnv
```

New:

```python
H1EnvCfg
H1Env
```

Why?

Because if both have the same name, Python will **confuse them during import**.

So we separate them:

```
HumanoidEnv → original environment
H1Env → new environment
```

---

# 5. Register the New Task

After creating the new environment, we must **register it with Gym**.

Inside:

```
__init__.py
```

we add:

```python
gym.register(
    id="Isaac-H1-Direct-v0",
    entry_point="isaaclab_tasks.direct.humanoid:H1Env",
```

This means:

```
Task name: Isaac-H1-Direct-v0
Environment class: H1Env
```

Now Isaac Lab can run:

```
Isaac-H1-Direct-v0
```

---

# 6. RL Agent Configurations

Inside registration we also specify RL configs:

```python
kwargs={
    "env_cfg_entry_point": H1EnvCfg,
    "rl_games_cfg_entry_point": "...",
    "rsl_rl_cfg_entry_point": "...",
}
```

Meaning:

```
Environment config → H1EnvCfg
Training algorithm → PPO configs
```

Important note from the tutorial:

> If your changes are small, you can reuse the old RL configs.

If the robot is very different:

```
create new RL configs
```

---

# 7. Changing the Robot

This is the **main modification**.

Original humanoid robot:

```python
from isaaclab_assets import HUMANOID_CFG

robot: ArticulationCfg = HUMANOID_CFG.replace(...)
```

---

New robot:

```python
from isaaclab_assets import H1_CFG

robot: ArticulationCfg = H1_CFG.replace(...)
```

Now the environment loads:

```
Unitree H1 robot
```

instead of

```
generic humanoid
```

---

# 8. What is `ArticulationCfg`?

`ArticulationCfg` describes a **robot with joints**.

Example robot structure:

```
robot
 ├─ hip
 ├─ knee
 ├─ ankle
 ├─ shoulder
 └─ elbow
```

This structure is called an **articulation**.

So:

```
ArticulationCfg = robot configuration
```

---

# 9. Updating `joint_gears`

Each robot joint has a **gear ratio**.

Example:

```
motor → gearbox → joint
```

The gear value affects:

```
torque strength
movement speed
```

Example list:

```python
joint_gears = [
50.0,  # left_hip_yaw
50.0,  # right_hip_yaw
50.0,  # torso
...
]
```

Why must we change it?

Because:

```
Humanoid robot joints ≠ H1 robot joints
```

The number and order are different.

---

# 10. Adjust Action Space

Original humanoid:

```python
action_space = 21
```

Meaning:

```
21 joints controlled
```

H1 robot:

```python
action_space = 19
```

Meaning:

```
19 joints controlled
```

So the RL policy outputs:

```
19 actions
```

instead of 21.

---

# 11. Adjust Observation Space

Original:

```
observation_space = 75
```

New robot:

```
observation_space = 69
```

Why?

Because observation contains things like:

```
joint positions
joint velocities
body orientation
contact forces
```

Different robot → different number of joints → different observation size.

---

# 12. Running Training

After modifying everything we can train:

```
./isaaclab.sh -p scripts/reinforcement_learning/rl_games/train.py \
--task Isaac-H1-Direct-v0 \
--headless
```

Meaning:

```
Train H1 robot using RL-Games PPO
```

`--headless` means:

```
no graphics
faster training
```

---

# 13. Visualizing the Result

After training:

```
./isaaclab.sh -p scripts/reinforcement_learning/rl_games/play.py \
--task Isaac-H1-Direct-v0 \
--num_envs 64
```

This opens simulation and shows:

```
H1 robot walking
```

---

# 14. Why This Method Is Useful

Instead of building everything:

```
robot
reward
physics
observations
termination
```

You reuse:

```
existing locomotion environment
```

and only change:

```
robot model
joint parameters
action/observation size
```

This saves **a huge amount of work**.

---

# 15. Important Warning from the Tutorial

Even if changes look small:

```
robot change
```

some hidden dependencies may break.

Examples:

```
reward depends on foot names
contact sensors
body indexes
joint ordering
```

So sometimes you must inspect:

```
reward function
observation function
termination condition
```

---

# 16. Simple Mental Model

Think of the environment like this:

```
LocomotionEnv
   │
   ├── Robot model
   ├── Physics
   ├── Observations
   ├── Rewards
   └── Actions
```

You replaced only:

```
Robot model
```

Everything else stays the same.

---

✅ **Final Summary**

Steps to modify an Isaac Lab environment:

1️⃣ Copy the existing environment file
2️⃣ Rename the classes
3️⃣ Register a new task
4️⃣ Replace the robot asset
5️⃣ Update joint parameters
6️⃣ Adjust action and observation sizes
7️⃣ Train the RL agent
