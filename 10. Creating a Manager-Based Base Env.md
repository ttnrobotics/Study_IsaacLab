This tutorial introduces one of the **most important concepts in Isaac Lab**:
👉 **Environment (Env)**.

I’ll explain it in a **simple, intuitive way**, because this structure is used in **robot control and reinforcement learning**.

---

# 1. What is an Environment?

An **environment** is a wrapper that organizes everything needed for a simulation.

Think of it like a **container that holds the robot world**.

Example:

```text
Environment
   │
   ├── Scene (robots, ground, lights)
   ├── Observations (what the robot can see)
   ├── Actions (how the robot can move)
   └── Events (reset, randomization, etc.)
```

Instead of manually controlling everything, you interact with **one object: the environment**.

---

# 2. Two Types of Environments in Isaac Lab

Isaac Lab provides two environment types.

### 1️⃣ `ManagerBasedEnv`

Used for **normal robot control**.

Example:

* test robot controller
* simulate robot behavior
* run physics experiments

No rewards or episode termination.

---

### 2️⃣ `ManagerBasedRLEnv`

Used for **reinforcement learning**.

Includes:

```text
Rewards
Terminations
Curriculum
Commands
```

Example:

```text
Robot learns to walk
Robot learns to balance
Robot learns to grasp
```

This tutorial uses **ManagerBasedEnv** (the simpler one).

---

# 3. What is inside a ManagerBasedEnv?

The environment uses several **managers**.

Think of them like **modules that control different parts of the simulation**.

```text
ManagerBasedEnv
     │
     ├── Scene Manager
     ├── Action Manager
     ├── Observation Manager
     └── Event Manager
```

Each one has a specific job.

---

# 4. Scene Manager

The **scene** contains everything in the world.

Example:

```text
Scene
  │
  ├── Ground
  ├── Light
  └── Cartpole robot
```

This uses the **InteractiveScene** we learned earlier.

So the environment contains:

```text
Environment
   │
   └── Scene
        ├── Ground
        ├── Light
        └── Cartpole
```

---

# 5. Action Manager

Actions define **how the robot can move**.

Example:

For the cartpole:

```text
Cart ← apply force
Pole ← balance
```

The action is **force applied to the cart**.

Code:

```python
joint_efforts = mdp.JointEffortActionCfg(
    asset_name="robot",
    joint_names=["slider_to_cart"],
    scale=5.0
)
```

Meaning:

```text
Apply force to the joint
slider_to_cart
```

And scale the force by **5**.

So actions look like:

```text
Agent action → force on cart
```

---

# 6. Observation Manager

Observations are **what the agent can see**.

Example observations:

```text
cart position
cart velocity
pole angle
pole angular velocity
```

These are the inputs for the controller or AI.

---

Observations are organized into **groups**.

Example:

```python
class PolicyCfg(ObsGroup):
```

The **policy group** contains observations used by the controller.

---

Inside the group we define observation terms.

Example:

```python
joint_pos_rel = ObsTerm(func=mdp.joint_pos_rel)
joint_vel_rel = ObsTerm(func=mdp.joint_vel_rel)
```

These compute:

```text
joint position
joint velocity
```

---

Then they are combined.

```python
self.concatenate_terms = True
```

Meaning:

Instead of:

```text
[pos]
[vel]
```

They become:

```text
[pos, vel]
```

One observation vector.

---

# 7. Event Manager

Events are operations triggered at certain times.

Examples:

```text
Reset robot
Randomize mass
Randomize friction
Change colors
```

Events help **create randomness**.

This is important for **robust robot training**.

---

There are three event types.

### 1️⃣ Startup

Runs **once when simulation starts**.

Example:

```text
randomize pole mass
```

Code:

```python
mode="startup"
```

---

### 2️⃣ Reset

Runs **when environment resets**.

Example:

```text
randomize robot position
```

Code:

```python
mode="reset"
```

---

### 3️⃣ Interval

Runs **periodically**.

Example:

```text
change lighting every 100 steps
```

Code:

```python
mode="interval"
```

---

# 8. Example Events in the Tutorial

### Randomize pole mass

```python
add_pole_mass
```

This adds random mass between:

```text
0.1 – 0.5 kg
```

Purpose:

```text
make simulation less predictable
```

---

### Reset cart position

Every reset:

```text
cart position = random(-1,1)
cart velocity = random(-0.1,0.1)
```

---

### Reset pole angle

Random initial pole angle:

```text
-22.5° to +22.5°
```

This makes the balancing task harder.

# 9. Environment Configuration

Now we combine everything.

```python
class CartpoleEnvCfg(ManagerBasedEnvCfg)
```

This config defines the **entire environment**.

---

Inside we specify:

```python
scene = CartpoleSceneCfg()
observations = ObservationsCfg()
actions = ActionsCfg()
events = EventCfg()
```

So the structure becomes:

```text
CartpoleEnv
    │
    ├── Scene
    ├── Observations
    ├── Actions
    └── Events
```

---

# 10. Simulation Settings

Inside `__post_init__` we set simulation parameters.

Example:

### Camera view

```python
self.viewer.eye = [4.5,0,6]
```

Camera position.

---

### Simulation timestep

```python
self.sim.dt = 0.005
```

Meaning:

```text
Physics step = 5 ms
```

Frequency:

```text
200 Hz physics
```

---

### Environment step frequency

```python
self.decimation = 4
```

Meaning:

```text
1 environment step
= 4 physics steps
```

So control runs at:

```text
200 Hz / 4 = 50 Hz
```

# 11. Running the Environment

The simulation loop becomes very simple.

Before (without env):

```python
reset robot
apply action
write to sim
step physics
update buffers
```

Now:

```python
env.reset()
env.step(action)
```

Much cleaner.

---

# 12. Example Simulation Loop

```python
env.reset()

while running:

    action = random()

    obs, info = env.step(action)
```

The environment automatically handles:

```text
Physics stepping
Action application
Observation calculation
Events
```

---

# 15. Why `torch.inference_mode()` is used

The simulation uses **PyTorch tensors**.

But we don't need gradients.

So we disable them:

```python
with torch.inference_mode():
```

Benefits:

```text
Faster simulation
Less memory
No gradient computation
```
