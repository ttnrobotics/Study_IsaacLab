I’ll explain this **in a very simple way** so you clearly understand the difference and how the code works. Think of it as **two ways to build an RL environment in Isaac Lab**.

---

# 1. Two Ways to Create RL Environments in Isaac Lab

Isaac Lab provides **two workflows**.

### 1️⃣ Manager-Based Workflow

You already saw this earlier.

You define many **configuration classes**:

```
ActionsCfg
ObservationsCfg
RewardsCfg
TerminationsCfg
EventsCfg
```

Then Isaac Lab **automatically connects them**.

So the structure is:

```
Config files
      ↓
Managers
      ↓
Environment
```

Good for:

* big research projects
* modular design
* reusable components

---

### 2️⃣ Direct Workflow

This tutorial shows **Direct Workflow**.

Instead of managers, **you write the functions yourself**.

Example:

```
def _get_rewards()
def _get_observations()
def _get_dones()
def _reset_idx()
```

So the structure becomes:

```
Environment class
   ├─ scene setup
   ├─ actions
   ├─ rewards
   ├─ observations
   ├─ reset
   └─ termination
```

Everything is in **one file**.

---

### Simple analogy

Manager-based:

```
Restaurant kitchen

Chef manager
Soup manager
Dessert manager
```

Direct workflow:

```
One chef cooks everything
```

---

# 2. What the DirectRLEnv Class Is

The main class is:

```
DirectRLEnv
```

You create your environment like this:

```python
class CartpoleEnv(DirectRLEnv):
```

This means:

```
CartpoleEnv inherits DirectRLEnv features
```

And you must implement functions such as:

| Function              | Purpose                |
| --------------------- | ---------------------- |
| `_setup_scene()`      | create the world       |
| `_pre_physics_step()` | process actions        |
| `_apply_action()`     | apply actions to robot |
| `_get_observations()` | compute observations   |
| `_get_rewards()`      | compute rewards        |
| `_get_dones()`        | check termination      |
| `_reset_idx()`        | reset environments     |

These are the **core RL components**.

---

# 3. Configuration Class

Just like manager workflow, there is still a **config class**.

```
CartpoleEnvCfg
```

Example:

```python
class CartpoleEnvCfg(DirectRLEnvCfg):
```

This stores environment settings.

---

### Important parameters

#### Action size

```
action_space = 1
```

Meaning:

```
agent outputs 1 value
```

Example:

```
force applied to cart
```

---

#### Observation size

```
observation_space = 4
```

The agent sees:

```
pole angle
pole velocity
cart position
cart velocity
```

Total = 4.

---

#### Episode length

```
episode_length_s = 5
```

Episode lasts **5 seconds**.

---

#### Number of environments

```
num_envs = 4096
```

Isaac Lab trains **4096 simulations at once** for fast RL training.

# 4. Scene Creation

Function:

```
_setup_scene()
```

This builds the simulation world.

Example code:

```python
self.cartpole = Articulation(self.cfg.robot_cfg)
```

This loads the **cartpole robot**.

Then:

```
spawn_ground_plane()
```

adds the floor.

---

### Cloning environments

```
self.scene.clone_environments()
```

This creates **thousands of parallel environments**.

Example:

```
Env 0 → cartpole
Env 1 → cartpole
Env 2 → cartpole
...
Env 4095 → cartpole
```

This is **massively parallel RL training**.

### Adding lights

```
DomeLight
```

Only for visualization.

# 5. Processing Actions

The agent outputs an action:

```
action = policy(observation)
```

But the environment must **convert this into robot commands**.

---

### Step 1 — Pre physics step

Function:

```
_pre_physics_step(actions)
```

Example:

```python
self.actions = self.action_scale * actions
```

Meaning:

```
scale action strength
```

Example:

```
policy output = 0.3
scale = 100

force = 30 N
```

---

### Step 2 — Apply action

Function:

```
_apply_action()
```

Example:

```python
self.cartpole.set_joint_effort_target(...)
```

Meaning:

```
apply force to cart joint
```

This moves the cart.

---

# 6. Observations

Function:

```
_get_observations()
```

This tells the **agent what it can see**.

Example:

```python
obs = torch.cat(...)
```

The observation vector is:

```
[pole_angle,
 pole_velocity,
 cart_position,
 cart_velocity]
```

Example observation:

```
[0.12, 0.03, 0.4, -0.1]
```

Meaning:

```
pole slightly tilted
cart moving left
```

The environment returns:

```
{"policy": obs}
```

The key **policy** is used by the RL algorithm.

---

# 7. Reward Function

Function:

```
_get_rewards()
```

Calls:

```
compute_rewards()
```

Reward formula:

```
reward =
    alive_reward
  + pole_angle_penalty
  + cart_velocity_penalty
  + pole_velocity_penalty
  + termination_penalty
```

---

### Example reward components

Alive reward:

```
+1 every step
```

Pole angle penalty:

```
-(pole_angle)^2
```

Cart velocity penalty:

```
-|cart_velocity|
```

Pole velocity penalty:

```
-|pole_velocity|
```

Termination penalty:

```
-2 if episode ends
```

---

### Why PyTorch JIT is used

```
@torch.jit.script
```

This compiles the function for **faster execution**.

Very important when running **thousands of environments**.

---

# 8. Termination Conditions

Function:

```
_get_dones()
```

Returns two things:

```
terminated
timeout
```

---

### Termination cases

#### 1. Cart too far

```
|cart_position| > 3
```

Episode ends.

---

#### 2. Pole falls

```
|pole_angle| > 90°
```

Episode ends.

---

#### 3. Time limit

```
episode_length reached
```

Episode resets.

---

# 9. Resetting Environments

Function:

```
_reset_idx()
```

This resets environments that finished.

Example:

```
Env 3 finished
Env 17 finished
Env 25 finished
```

Only these environments reset.

---

### Randomizing pole start

```
initial_pole_angle_range
```

Example:

```
[-0.25, 0.25] rad
```

Meaning the pole starts slightly tilted.

This improves learning.

---

# 10. Domain Randomization

To make policies **robust**, Isaac Lab can randomize physics.

Examples:

Random friction:

```
0.7 → 1.3
```

Random actuator gains.

Random gravity.

Example:

```
gravity = 9.81 ± noise
```

This helps **sim-to-real transfer**.

---

# 11. Noise Injection

You can also add noise to:

### Actions

Example:

```
action + Gaussian noise
```

---

### Observations

Example:

```
sensor noise
```

This prevents **overfitting to simulation**.

# 13. Manager vs Direct Workflow (Important)

| Feature          | Manager Workflow | Direct Workflow |
| ---------------- | ---------------- | --------------- |
| Code structure   | modular          | single script   |
| abstraction      | high             | low             |
| flexibility      | medium           | high            |
| easier debugging | harder           | easier          |
| recommended for  | large projects   | custom tasks    |

---

### Real research usage

Manager workflow → large robotics tasks.

Direct workflow → when you want **full control**.
