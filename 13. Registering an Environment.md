I’ll explain this **step-by-step in a very simple way**. Think of this tutorial as explaining **how Isaac Lab creates and runs many RL environments easily using Gymnasium**.

---

# 1. The Problem Before (Old Way)

In the previous tutorial, you created the environment **manually** like this:

```python
env_cfg = CartpoleEnvCfg()
env_cfg.scene.num_envs = args_cli.num_envs
env_cfg.sim.device = args_cli.device

env = ManagerBasedRLEnv(cfg=env_cfg)
```

### What happens here?

You manually:

1️⃣ Create the configuration
2️⃣ Set number of environments
3️⃣ Set device (CPU/GPU)
4️⃣ Create the environment object

This works, but it becomes **hard to manage when you have many environments**.

Example:

* Cartpole
* Quadruped
* Manipulator
* Drone
* Camera environments

You would need to import and configure each one manually.

❌ **Not scalable**

---

# 2. The Better Way: Gym Registry

Instead, Isaac Lab uses **Gymnasium's environment registry**.

This means:

You **register environments once**, and then create them easily using:

```python
env = gym.make("Environment-Name")
```

Exactly like standard reinforcement learning libraries.

Example:

```python
env = gym.make("Isaac-Cartpole-v0")
```

Much simpler.

---

# 3. Environment Registration

Before we can call `gym.make()`, we must **register the environment**.

Example:

```python
gym.register(
    id="Isaac-Cartpole-v0",
    entry_point="isaaclab.envs:ManagerBasedRLEnv",
    kwargs={
        "env_cfg_entry_point": "...CartpoleEnvCfg"
    },
)
```

### What this means

This tells Gym:

> "If someone asks for `Isaac-Cartpole-v0`, create this environment."

---

### Important fields

### 1️⃣ id

Environment name.

```
Isaac-Cartpole-v0
```

Naming convention:

```
Isaac-Task-Robot-Version
```

Examples:

```
Isaac-Cartpole-v0
Isaac-Velocity-Anymal-C-v0
Isaac-Cartpole-RGB-v0
```

---

### 2️⃣ entry_point

This tells Gym **which Python class to create**.

Example:

```
entry_point="isaaclab.envs:ManagerBasedRLEnv"
```

Meaning:

```
module: isaaclab.envs
class: ManagerBasedRLEnv
```

So Gym will internally do something like:

```python
env = ManagerBasedRLEnv(...)
```

---

### 3️⃣ kwargs

Extra parameters to configure the environment.

Example:

```python
kwargs={
 "env_cfg_entry_point": "...CartpoleEnvCfg"
}
```

This tells Isaac Lab:

> Use `CartpoleEnvCfg` as the default configuration.

---

# 4. Why `import isaaclab_tasks` Is Important

In the script you see:

```python
import isaaclab_tasks
```

This line **registers all environments automatically**.

Because inside `isaaclab_tasks/__init__.py`:

```
gym.register(...)
gym.register(...)
gym.register(...)
```

So importing the module automatically registers environments like:

```
Isaac-Cartpole-v0
Isaac-Cartpole-RGB-v0
Isaac-Cartpole-Depth-v0
```

# 5. Creating the Environment

Now the script creates the environment.

### Step 1 — Parse configuration

```python
env_cfg = parse_env_cfg(
    args_cli.task,
    device=args_cli.device,
    num_envs=args_cli.num_envs
)
```

Example input:

```
--task Isaac-Cartpole-v0
--num_envs 32
--device cuda
```

This function:

✔ loads default config
✔ modifies device
✔ modifies number of environments

---

### Step 2 — Create environment

```python
env = gym.make(args_cli.task, cfg=env_cfg)
```

If:

```
args_cli.task = "Isaac-Cartpole-v0"
```

Then Gym:

1️⃣ finds registered environment
2️⃣ loads `ManagerBasedRLEnv`
3️⃣ loads `CartpoleEnvCfg`
4️⃣ builds the simulation

---

# 6. What Is a Vectorized Environment?

Important concept.

Isaac Lab runs **many environments at the same time**.

Example:

```
num_envs = 32
```

Then Isaac Lab runs:

```
32 cartpoles simultaneously
```

Like this:

```
Cartpole 1
Cartpole 2
Cartpole 3
...
Cartpole 32
```

Why?

Because **RL training needs many samples**.

This greatly speeds up training.

# 8. Reset the Environment

```python
env.reset()
```

This resets all environments.

Example:

```
32 cartpoles reset
```

---

# 9. Simulation Loop

Main loop:

```python
while simulation_app.is_running():
```

Meaning:

Run simulation until the window closes.

---

# 10. Random Actions

Inside the loop:

```python
actions = 2 * torch.rand(env.action_space.shape) - 1
```

This generates random numbers between:

```
-1 and 1
```

Example:

```
[-0.5]
[0.8]
[-0.1]
```

These are **random control commands**.

---

# 11. Apply the Action

```python
env.step(actions)
```

This does:

1️⃣ Apply action to robot
2️⃣ Run physics simulation
3️⃣ Compute observation
4️⃣ Compute reward
5️⃣ Check done condition

Exactly like standard Gym.

---

# 12. Torch Inference Mode

```python
with torch.inference_mode():
```

This disables gradients.

Why?

Because this script **is not training**.

It only runs random actions.

So gradients are unnecessary.

This saves memory and speeds up execution.

---

# 13. Closing the Environment

When simulation ends:

```python
env.close()
```

This:

✔ closes physics simulation
✔ releases GPU memory
✔ shuts down Isaac Sim

# 16. Manager-Based vs Direct Environments

Two types exist.

### Manager-based

```
ManagerBasedRLEnv
```

Uses modular managers:

```
Observation manager
Reward manager
Action manager
Termination manager
```

Better for complex RL environments.

---

### Direct environments

```
CartpoleEnv
```

You manually implement everything.

Used when you want full control.

---

# 17. Big Picture

The workflow becomes:

```
1 Register environments (once)

2 Load environment using gym.make()

3 Reset environment

4 Loop:
     action = policy(obs)
     obs, reward, done = env.step(action)

5 Train RL agent
```

Exactly the same as **OpenAI Gym RL workflow**.
